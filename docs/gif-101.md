# GIF 101 — How It Works

A technical walkthrough of the codebase: what gif does, how it's structured, and the order of execution for a tool call.

---

## What it is, physically

Two Docker containers: a PostgreSQL 16 database and a Node.js MCP server. No message broker, no cache, no external services. Everything is in the database.

```
Client (AI model / test script)
    ↓  HTTP POST to :3100/mcp
MCP Server (mcp-server/src/index.ts)
    ↓  validates persona, creates session, dispatches tool
PostgreSQL (gif schema)
    ↓  personas, sessions, audit_events, scope_violations
```

---

## The three concepts you need

**Persona** — a governance identity for an AI agent. Not a user, not a service account. It carries a declared `purpose` (required — non-nullable, schema constraint), a `scope_definition` (JSONB: what tools and sources are permitted), temporal bounds (`valid_from`, `valid_until`), and an optional parent persona for delegation chains. Personas are created by human administrators before any AI action occurs.

**Session** — one tool call, wrapped. When an AI calls a tool, gif creates a session row (INSERT), runs the tool, logs the audit event, then closes the session (UPDATE `ended_at`). Sessions are not persistent login windows — they are microseconds-to-seconds governance envelopes around a single invocation.

**Audit event** — the immutable record. Every tool call produces a row in `audit_events`: who (persona), what session, which tool, what outcome (`allowed` / `denied` / `deferred` / `error`), sources touched, purpose declared. `gif_app` (the runtime role) has INSERT-only on this table — no UPDATE, no DELETE at the database permission level.

## Order of execution for a single tool call

`index.ts` is the chokepoint. Every tool call follows this sequence:

```
1. Client sends CallTool { name, arguments: { persona_id, ... } }

2. TOOL_REGISTRY.get(name) → ToolHandler  (unknown tool → error returned)

3. validatePersona(persona_id)
   → SELECT from gif.personas
   → checks: exists, status = 'active', valid_from ≤ now ≤ valid_until,
             governance_review_status = 'approved'
   → failure: return error, nothing logged

4. If skipSession = true (persona_validate only):
   → execute tool, return result — no session, no audit

5. Otherwise:
   createSession(personaId, invocationContext)
   → INSERT INTO sessions, returns session_id UUID

6. toolHandler.execute(args, persona, sessionId)
   → tool runs its logic
   → may invoke checkCombinationPolicies (aggregation risk; adopter-invoked
     per GIF-012 framework boundary)
   → may log scope violations internally
   → returns ToolResult { content, isError }

7. logAuditEvent({ personaId, sessionId, toolName, outcome, ... })
   → INSERT INTO audit_events
   → never throws — audit failure cannot mask tool response

8. closeSession(sessionId)
   → UPDATE sessions SET ended_at = now()
   → never throws

9. Return ToolResult to client
```

Steps 7–8 run in a `finally` block — they execute whether the tool succeeded or threw.

## The tool registry

`tools/registry.ts` exports a `Map<toolName, ToolHandler>`. The enforcement engine has zero knowledge of which tools exist or what they do — it only knows the `ToolHandler` interface. Adding a tool means adding it to the map.

**One handler per action.** Each discrete operation the AI is allowed to perform is a separate handler — `app_search`, `app_create_entry`, `app_delete` are three handlers, not one handler with an `action` parameter. This is not convention — it is what makes Persona scope enforcement meaningful at the action level. If two operations share a handler, you cannot scope a Persona to one without the other.

**Secrets.** Handler code accesses application credentials via `process.env`. API keys, tokens, and any other secrets required by a handler belong in environment variables — not in the GIF database. For development, add them to `.env` alongside the GIF credentials. For production, they come from the adopter's secret management infrastructure (AWS Secrets Manager, Vault, Kubernetes Secrets, etc.) — injected as environment variables when the container starts. GIF does not manage application credentials; that is the adopter's operational concern.

**`db_read.ts` and `db_write.ts` are reference adopter tools**, not GIF framework tools. They exist to demonstrate the handler pattern and can be read as templates. Adopter implementations follow the same shape with their own application logic in place of the gif table queries.

Current tools (5 active):

| Tool | Purpose | skipSession |
|------|---------|------------|
| `persona_validate` | Check if a persona is valid | yes — no audit |
| `persona_create` | Create a new persona | no — first-class audit event |
| `persona_revoke` | Revoke or suspend a persona | no — terminates active sessions |
| `db_read` | Reference adopter tool — query gif tables (allowlisted); demonstrates the handler pattern | no |
| `db_write` | Reference adopter tool — write to gif tables; demonstrates the handler pattern | no |

## The enforcement package

`enforcement.ts` exports `createEnforcement(pool)` — a factory that returns enforcement functions bound to the injected PostgreSQL pool. This is what adopter servers import. They inject their own `pg.Pool`; gif does not own the connection.

Key functions:

- **`validatePersona(personaId)`** — checks existence, active status, temporal bounds
- **`createSession(personaId, invocationContext)`** — INSERT into sessions, returns session_id
- **`closeSession(sessionId)`** — UPDATE sessions.ended_at; never throws
- **`logAuditEvent(params)`** — INSERT into audit_events; never throws
- **`logScopeViolation(params)`** — INSERT into scope_violations; never throws
- **`checkCombinationPolicies(sessionId, personaId, incomingSourceRefs)`** — pre-execution aggregation risk check; evaluates active policies with first-match resolution (v0.1; exhaustive evaluation is a v0.2 trajectory item); fails closed on DB error. Caller-declared `incomingSourceRefs` is trusted by this v0.1 evaluator — runtime declaration validation is a v0.2 trajectory item

## Database schema at a glance

| Table | Purpose | gif_app grant |
|-------|---------|---------------|
| `personas` | Governance identities | SELECT, INSERT — `admin_read` permission required to query via MCP tools |
| `sessions` | Per-invocation envelopes | SELECT, INSERT, UPDATE |
| `audit_events` | Immutable action log | INSERT only |
| `scope_violations` | Out-of-scope attempt records | INSERT only |
| `delegation_chain` | Parent→child persona relationships | SELECT, INSERT |
| `revocation_log` | Persona state change history | SELECT, INSERT |
| `tool_registry` | Capability inventory | SELECT |
| `combination_policies` | Aggregation risk rules | SELECT |
| `user_persona_assignments` | External user → persona bindings | SELECT, INSERT, UPDATE |
| `schema_migrations` | Applied migration tracking | SELECT, INSERT |

All tables live in the `gif` schema. `gif_admin` owns all objects and runs migrations. `gif_app` is the runtime role — it cannot DELETE or UPDATE audit tables.

## Adopter accountability obligations

GIF records what happened, under which persona, and when — as a structural guarantee. What GIF cannot capture automatically is the human and system context behind the call: which human account the AI is running under, which product session triggered it, which environment it ran in. This is the adopter's responsibility.

**`responsible_human_id` must be present in `invocation_context` on every invocation.** Without it, the audit trail has a runtime accountability gap — GIF can name the provisioning human (structurally recorded at persona creation) but not the human whose account the AI was operating under at runtime.

For service account or background job invocations where no user is active, pass a namespaced system identifier. An absent field is not traceable; a namespaced string is.

The full contract — required fields, recommended fields, and patterns for service accounts, sub-agents, and product sessions — is in [`docs/adopter-invocation-context.md`](adopter-invocation-context.md).

**Note on `persona_id` as a bearer token:** GIF validates that a persona is active and in scope but does not structurally verify that the calling process is the authorized holder of the `persona_id`. This is a deliberate, documented constraint (GIF-014). The mitigation: `personas` is gated behind `admin_read` (not enumerable), every persona has a named provisioner, and the audit trail is append-only. Adopters fulfilling the `responsible_human_id` obligation complete the accountability chain.

## How to get output

**Start the stack:**
```bash
cp .env.example .env        # fill in passwords
docker compose up -d --build
docker compose logs postgres # watch init — look for "GIF Init Complete"
```

**Smoke test:**
```bash
curl -s http://localhost:3100/health
# → {"status":"ok","service":"gif-mcp-server"}
```

**Run the full integration suite (8 test files, real PostgreSQL — no functional mocks; one error-injection mock in `test_combination_policies` Test 8 to exercise the fail-closed path):**
```bash
cd mcp-server
npm install && npm run build
source ../.env
PGHOST=localhost PGPORT=$PGPORT_HOST \
PGPASSWORD=$GIF_APP_PASSWORD PGADMINPASSWORD=$GIF_ADMIN_PASSWORD \
IDENTITY_HMAC_SECRET=$IDENTITY_HMAC_SECRET \
npm test
```

Tests connect directly to the postgres container, create personas, call tools over HTTP to the MCP server, and verify the audit trail was written correctly. No test framework — plain Node.js `.mjs` files with manual assertions.

**Reset to a clean state:**
```bash
docker compose down -v      # drops postgres_data volume
docker compose up -d        # fresh init, all 12 migrations reapply
```
