# Migrating from v0.1 to v0.2

> **Status:** v0.2 design pass complete (GIF-018 through GIF-022 merged); v0.2 implementation landed for `session_start` / `session_close` tools and the dispatcher cutover on `main`. Treat the contract below as the **target** for the `v0.2.0-rc.1` tag. Wire-level error code names are RC-period detail per GIF-020 §What This ADR Does Not Decide and GIF-022 §I2; everything else listed below is locked.

## Why v0.2 is a Breaking Release

The MCP 2026-07-28 Specification Release Candidate removes the `initialize` handshake and `Mcp-Session-Id` from the protocol core (SEP-2575 + SEP-2567). GIF v0.1's combination-policy enforcement accumulates sensitivity across calls within an MCP transport session that no longer exists in v0.2 of the spec. v0.2 introduces an explicit governance session handle minted and tracked by GIF.

The architectural decisions are recorded in:

- [GIF-018](../../decisions/GIF-018-stateless-mcp-session-handles.md) — scoping ADR (forces, design space, migration constraints).
- [GIF-019](../../decisions/GIF-019-session-handle-mint-and-propagation.md) — mint point (`session_start` MCP tool) and propagation channel (explicit `gif_session_id` argument on every governed tool's input schema).
- [GIF-020](../../decisions/GIF-020-session-closure-semantics.md) — closure semantics: caller-close (`session_close` MCP tool) + hard wall-clock TTL (`GIF_SESSION_TTL_SECONDS`, default 86400), lazy expiry, no inactivity timeout in v0.2.0.
- [GIF-021](../../decisions/GIF-021-session-schema-repurpose.md) — schema impact: `gif.sessions` repurposed in place via a comment-only migration; no DDL changes, no row purge, no FK migration.
- [GIF-022](../../decisions/GIF-022-conformance-surface.md) — conformance surface: six categories of MUST behaviors (C1.x mint, C2.x dispatch validation, C3.x closure, C4.x audit emission, C5.x combination scoping, C6.x identity gates) plus ten implementation-defined items (I1–I10).

## Breaking Changes

### Session handle is now explicit

**v0.1 behavior.** The dispatcher minted a fresh `sessions` row at the start of every governed call and closed it (`ended_at = now()`) in the same handler's `finally` block. The row's lifetime was one tool call; there was no caller-visible session identifier. v0.1's combination-policy mechanism (GIF-011) was specified to accumulate sources across calls sharing a `session_id`, but because the dispatcher recycled the row per call, the cross-call accumulation query at `mcp-server/src/enforcement.ts` always returned the empty set in production. The mechanism was structurally inert.

**v0.2 behavior.** The governance session is now a named, caller-threaded handle (`gif_session_id`) that spans many tool calls. It is minted by a new MCP tool (`session_start`), passed as an explicit `gif_session_id` argument on every governed tool's input schema, and ended either by an explicit `session_close` call or by hard wall-clock TTL expiry on the next governed call after `started_at + GIF_SESSION_TTL_SECONDS`. The dispatcher no longer mints or closes sessions implicitly. Combination-policy accumulation is now load-bearing for the first time: sources accumulate across all calls sharing a single `gif_session_id` and reset cleanly across distinct handles (GIF-022 C5.1/C5.2).

**Migration steps for an adopter MCP server pinning to the v0.2 tag of `gif-enforcement`:**

1. **Repin the git dependency** in `package.json` from the `v0.1.x` tag to the `v0.2.0` tag (or `v0.2.0-rc.N` during the RC window). Per GIF-007 the dependency is tag-pinned over SSH.
2. **Update SDK imports** from the single `@modelcontextprotocol/sdk@^1.x` package to the split `2.0.0` layout (`@modelcontextprotocol/server` + `@modelcontextprotocol/node` for adopters running plain Node http, plus the `@cfworker/json-schema` peer dep). The full import map, peer-dep gotcha, and ESM requirement are in "MCP SDK package layout and version" below.
3. **Add a `session_start` call** at the beginning of each logical work session in your host code. Persist the returned `gif_session_id` for the lifetime of that logical session.
4. **Thread `gif_session_id`** as an arg on every subsequent governed tool call (`db_read`, `db_write`, `persona_create`, `persona_revoke`, and any adopter-defined governed tools). The arg is required by `inputSchema`; calls that omit it are rejected before the tool handler runs.
5. **Add a `session_close` call** at logical session end (recommended). Adopters that prefer to let sessions age out under TTL may skip the explicit close — the audit trail will reflect which pattern was used (GIF-020 §Audit-Trail Story, Paths 1–3).
6. **Handle the new error conditions** on governed calls: `SESSION_NOT_FOUND`, `SESSION_PERSONA_MISMATCH`, `SESSION_CLOSED`, `SESSION_EXPIRED`. Recommended caller response is to call `session_start` again and re-thread the new handle. Wire-level error code names ship with `v0.2.0-rc.1` and may be revised in RC feedback (GIF-022 I2); audit event types are locked.
7. **Update smoke tests and integration tests** to call `session_start` before exercising governed tools. Tests that relied on v0.1's silent per-call session minting will not run on v0.2 deployments — per GIF-018 M4 there is no silent v0.1 fallback.
8. **Set `GIF_SESSION_TTL_SECONDS`** in the deployment environment if the 24-hour default does not fit the workflow profile. See "Adopter tool-call contract" below for the contract.

### Adopter tool-call contract

The v0.2 governance session is the surface every adopter MCP server interacts with. This subsection enumerates the wire-level pieces an adopter must implement (or, where a primitive is exported from `gif-enforcement`, must call). All MUSTs are sourced from GIF-019/020/021/022; this is a runbook, not a re-derivation.

#### New MCP tools: `session_start` and `session_close`

Both ship in the GIF enforcement engine's tool registry under "GIF framework tools" and are exposed through `tools/list` on any MCP server that imports `gif-enforcement`. Adopters do not implement them — adopters expose them to their callers.

`session_start` — mint a governance session handle.

```
Input:
  persona_id          string   required   UUID of an approved, active persona
  invocation_context  object   optional   Free-form adopter-supplied metadata
                                          (e.g., {"workflow": "intake"}).
                                          Captured at mint, immutable.

Output:
  gif_session_id      string              UUID identifying the new session.

Behavior:
  1. Validates persona via the existing _validatePersona path (returns
     PersonaInvalidReason on failure). Governance-review status must be
     'approved' per CLAUDE.md non-negotiable.
  2. INSERTs a row into gif.sessions with persona_id and invocation_context.
  3. INSERTs a session_start audit event linked to the new session_id.
  4. Returns the gif_session_id in the tool response.

Marked skipSession=true. The handler mints its own governance session;
the dispatcher does not pre-validate a handle for this call.
```

`session_close` — explicitly end a governance session.

```
Input:
  persona_id          string   required   UUID of the persona that owns the
                                          session.
  gif_session_id      string   required   Handle returned by session_start.

Output:
  closed              boolean             true on successful close.

Behavior:
  1. Validates persona via _validatePersona.
  2. SELECTs the sessions row by gif_session_id.
  3. Rejects if the session does not exist, is owned by a different persona,
     or is already closed (SESSION_NOT_FOUND / SESSION_PERSONA_MISMATCH /
     SESSION_ALREADY_CLOSED).
  4. On accept: sets sessions.ended_at = now() AND emits a session_close audit
     event linked to the session_id. Both writes are part of the
     successful-close commitment (GIF-022 C3.4).
  5. On reject: emits a session_rejected_closed audit event recording the
     attempt (source_ref carries the rejection code).

Marked skipSession=true. session_close is idempotent-by-rejection: a second
close on an already-closed session is rejected, not treated as a no-op.
The rejection itself emits an audit event.
```

A conformance harness can discover both tools through `tools/list`, exercise them, and observe rejections via response shape (GIF-022 C1.1, C3.1).

#### `gif_session_id` required on every governed tool

Every tool whose handler today receives a non-empty `sessionId` parameter via dispatch — operationally, every tool not flagged `skipSession=true` — gains a `gif_session_id` field of type string in its `inputSchema.properties` and in its `inputSchema.required` array. In the v0.2 reference implementation this is the set: `db_read`, `db_write`, `persona_create`, `persona_revoke`. Adopters that register additional governed tools through the registry pattern in `mcp-server/src/tools/registry.ts` must follow the same shape — adding `gif_session_id` to both `properties` and `required` — or their tools will be unreachable (the dispatcher rejects calls that omit the handle before reaching the handler).

Tools currently flagged `skipSession=true` (`persona_validate`, `session_start`, `session_close`) are exempted from this requirement and remain so. They either need no governance session (`persona_validate`) or operate on a caller-supplied handle directly (`session_close`) or mint their own (`session_start`).

#### New environment variable: `GIF_SESSION_TTL_SECONDS`

```
Name:      GIF_SESSION_TTL_SECONDS
Default:   86400 (24 hours)
Units:     seconds (positive integer)
Scope:     deployment-wide; same TTL applies to every persona
Read at:   process startup; reread does not happen
Validation: parsed via parseInt(..., 10); fails-fast on non-finite or
            non-positive values
```

The variable joins the env-var contract documented in `docs/secrets.md` (per GIF-017) as a non-secret configuration value. Deployments needing a different TTL set it at process start; deployments accepting the default omit it. The GIF MCP server logs the effective value at startup (`[server] GIF_SESSION_TTL_SECONDS=<value>`).

Per-persona TTL is deliberately deferred (GIF-020 §C5). If adopter signal warrants it, a future ADR can add a `personas.session_ttl_seconds` override column additively — NULL means "use the deployment default," presence means "override." v0.2.0 does not ship that surface.

#### Dispatcher rejection codes

The dispatcher validates the caller-supplied `gif_session_id` against ownership, closure, and TTL before executing any governed tool handler. The four rejection conditions returned in the response body's `reason` field are:

| Reason | Condition |
|---|---|
| `SESSION_NOT_FOUND` | No `sessions` row matches the presented `gif_session_id` (or the field was empty/whitespace). |
| `SESSION_PERSONA_MISMATCH` | A row exists but its `persona_id` does not match the asserted caller `persona_id`. |
| `SESSION_CLOSED` | The row exists and `sessions.ended_at IS NOT NULL`. Closure precedence applies — `SESSION_CLOSED` wins over `SESSION_EXPIRED` when both would apply. |
| `SESSION_EXPIRED` | The row exists, has not been closed, and `now() > sessions.started_at + GIF_SESSION_TTL_SECONDS`. |

The response shape mirrors persona validation:

```json
{
  "valid":   false,
  "reason":  "SESSION_EXPIRED",
  "message": "Session 4f9c... has expired (started_at: 2026-05-24T12:00:00Z, TTL: 86400s)"
}
```

…with `isError: true` set on the MCP tool result.

A fifth internal code, `SESSION_DB_ERROR`, exists for the case where the validation query itself fails (e.g., transient database unavailability). This is a fail-closed mode: the caller receives an error response, but **no audit event is emitted** for this branch (the implementation does not know the session's state and must not claim it in the audit trail). `SESSION_DB_ERROR` is not part of the wire-level contract a conformance harness should assert against — it is an implementation-internal fail-closed signal, mentioned here so adopters whose dispatchers call `validateSessionHandle` (see below) understand the rejection-side branch.

Wire-level error code names are part of the `v0.2.0-rc.1` implementation surface and may be revised in RC feedback (GIF-020 §What This ADR Does Not Decide, GIF-022 §I2). Adopters that branch on error code names should do so against the v0.2 tag they pinned to and refresh on each tag bump until `v0.2.0` final.

#### Audit-event mapping on rejection

Audit emission on the rejection path is **best-effort** — a failed INSERT does not change the response the caller receives (audit-never-throws, CLAUDE.md non-negotiable, GIF-022 C2.7, C4.4). The reference dispatcher emits:

| Rejection | Audit event type | `source_ref` | `session_id` on the audit row |
|---|---|---|---|
| `SESSION_NOT_FOUND` | `session_rejected_closed` | `SESSION_NOT_FOUND` | NULL (no row exists to link to) |
| `SESSION_PERSONA_MISMATCH` | `session_rejected_closed` | `SESSION_PERSONA_MISMATCH` | the presented handle |
| `SESSION_CLOSED` | `session_rejected_closed` | `SESSION_CLOSED` | the presented handle |
| `SESSION_EXPIRED` | `session_expired` | `SESSION_EXPIRED` | the presented handle |
| `SESSION_DB_ERROR` | *(none — see above)* | *(n/a)* | *(n/a)* |

Three of the four rejection branches collapse to `session_rejected_closed`; `source_ref` distinguishes them so a conformance harness or auditor querying `audit_events` can recover the exact condition without parsing the human-readable message. `session_expired` is its own event type because TTL expiry is the only rejection that can transition a session from "usable" to "rejected" without any state change on the session row itself (GIF-020 §Path 2).

The `session_close` tool's own rejection branches (when a caller tries to close an unknown or mismatched or already-closed session) emit `session_rejected_closed` with `source_ref` ∈ `{SESSION_NOT_FOUND, SESSION_PERSONA_MISMATCH, SESSION_ALREADY_CLOSED}`. The audit-event-type space is shared with dispatcher rejections by design; the toolName column on `audit_events` distinguishes the source.

Note that `audit_events.session_id` is nullable specifically to support the `SESSION_NOT_FOUND` case — recording the absent handle as a non-null value would misleadingly link the audit row to a session that does not exist. No schema migration was required; the column was already declared `UUID REFERENCES sessions(session_id)` without `NOT NULL`.

#### Dispatcher no longer auto-closes sessions

In v0.1, the dispatcher's `finally` block called `_closeSession()` (`sessions.ended_at = now()`) on every governed tool call. In v0.2 this call is **removed**. Closure is exclusively caller-driven (via `session_close`) or TTL-driven (via lazy expiry on the next governed call after `started_at + TTL`).

The consequence is a path the audit trail does not directly record: a session minted via `session_start`, used for one or more calls, and then never touched again will have its `sessions.ended_at` remain NULL forever; the effective expiry at `started_at + GIF_SESSION_TTL_SECONDS` produces **zero closure audit events** because no governed call ever lands to trigger the lazy `session_expired` emission (GIF-020 §Path 3, "silent dormancy"). This is by deliberate design under F5 (stateless dispatch — no background sweeper).

**Audit-trail reconstruction for silent dormancy.** Auditors and downstream integrations must not interpret `sessions.ended_at IS NULL` as "session still active." The correct reconstruction is:

```
effective_state(session_row) =
  if ended_at IS NOT NULL                                    → closed at ended_at
  elsif EXISTS session_expired audit_event for session_id    → expired at audit row's created_at
  elsif now() > started_at + GIF_SESSION_TTL_SECONDS         → dormant; expired at started_at + TTL
  else                                                       → active
```

The TTL value an auditor uses is the deployment's `GIF_SESSION_TTL_SECONDS` at the time the audit is performed. If a deployment changed the TTL between mint and audit, the auditor must apply the TTL in effect at the time of any subsequent governed call (or, for purely dormant sessions, the TTL in effect at the time of mint — recorded operationally, not in the schema). This is a known operational concern; v0.2.0 does not capture historical TTL values in the schema.

#### `validateSessionHandle` — adopter-contract primitive

Adopters who build their own MCP server (importing `gif-enforcement` per GIF-007) and dispatch their own governed tools call `validateSessionHandle` from the dispatcher to perform the session-handle check. It is exported by `createEnforcement(pool)` and ships as a first-class adopter-contract surface, alongside `validatePersona`, `createSession`, `logAuditEvent`, and `checkCombinationPolicies`. Adopters **must not** re-implement the closure-vs-expired classification themselves; closure precedence and rejection-reason taxonomy live in the primitive so that conformance is structural rather than reliant on each adopter dispatcher getting it right.

Signature:

```ts
validateSessionHandle(params: {
  personaId:    string;   // asserted caller persona_id
  gifSessionId: string;   // handle from the tool args
  ttlSeconds:   number;   // resolved at process startup from GIF_SESSION_TTL_SECONDS
}): Promise<SessionHandleValidationResult>

type SessionHandleValidationResult =
  | { valid: true;  sessionId: string }
  | {
      valid:          false;
      reason:         SessionRejectionReason;   // see table above
      message:        string;                   // human-readable
      auditSessionId: string | null;            // session_id to record on the
                                                // rejection-side audit event;
                                                // null on NOT_FOUND/DB_ERROR
    };

type SessionRejectionReason =
  | 'SESSION_NOT_FOUND'
  | 'SESSION_PERSONA_MISMATCH'
  | 'SESSION_CLOSED'
  | 'SESSION_EXPIRED'
  | 'SESSION_DB_ERROR';
```

Call shape (canonical dispatcher flow):

```ts
const sessionCheck = await validateSessionHandle({
  personaId:    persona_id,
  gifSessionId: args.gif_session_id,
  ttlSeconds:   GIF_SESSION_TTL_SECONDS,
});

if (!sessionCheck.valid) {
  if (sessionCheck.reason !== 'SESSION_DB_ERROR') {
    const eventType =
      sessionCheck.reason === 'SESSION_EXPIRED'
        ? 'session_expired'
        : 'session_rejected_closed';
    await logAuditEvent({
      personaId:       persona_id,
      sessionId:       sessionCheck.auditSessionId,
      eventType,
      toolName:        tool_name,
      outcome:         'rejected',
      sourceRef:       sessionCheck.reason,
      purposeDeclared: validation.persona.purpose,
    });
  }
  return /* error response with { valid: false, reason, message }, isError: true */;
}

// sessionCheck.valid === true; use sessionCheck.sessionId for the rest of dispatch.
```

The primitive does a single primary-key SELECT on `gif.sessions`; the dispatcher's hot path adds one round-trip on top of the existing persona validation (which adopters were already doing). `auditSessionId` is provided as a field on the rejection result so the caller composes the audit event with the correct `session_id` (NULL when the row does not exist, the handle otherwise) without having to re-derive the rule.

Adopters that ignore `validateSessionHandle` and write their own dispatcher-side check are not conformant per GIF-022 (the rejection taxonomy and closure precedence are MUSTs). The recommendation is unambiguous: import the primitive.

### Schema changes

- **New migrations:** `schema/013_session_v2_semantics.sql` — comment-only. Updates Postgres system-catalog comments on `gif.sessions` and on the `started_at` / `ended_at` / `invocation_context` columns to reflect v0.2 semantics per GIF-019 and GIF-020. **No DDL changes. No row purge. No FK migration.** Migration 013 is idempotent (`COMMENT ON` overwrites prior comments) and applies as part of the standard `init-db.sh` flow tracked by `schema/012_schema_migrations.sql`.
- **Forward migration runbook:** Apply migration 013 alongside the other v0.2 migrations during the standard `init-db.sh` pass. No downtime, no table rewrite, no row migration. Operators running ad-hoc psql can apply it directly (`\i schema/013_session_v2_semantics.sql`).
- **Migration 016 (issue #38, pre-014 adopters only):** Migration 014 added `gif.audit_events.canon_version TEXT NOT NULL DEFAULT 'gif-audit/1'`. On a clean install this is correct. For an adopter who recorded audit rows under v0.1 (hashed by the migration-006 pipe-delimited preimage) *before* applying 014, the `NOT NULL DEFAULT` retroactively stamps those historical rows `gif-audit/1`, a canon they were never hashed under, and the chain verifier then reports them as tamper (`mismatches`) instead of `uncheckable`. Migration 016 repairs this: it re-stamps to the legacy marker `gif-audit/0` only the rows it can cryptographically prove were hashed under migration 006 (stored `event_hash` equals the recomputed 006 pipe-form). The verifier does not recognize `gif-audit/0` and so classifies those rows as `uncheckable` (informational, never tamper). Genuinely tampered historical rows match neither preimage and stay flagged, so tamper detection is preserved, not masked. Idempotent and a no-op on clean installs. `canon_version` is not part of the hashed preimage, so no `event_hash` or chain linkage changes. Apply alongside the other migrations in the standard `init-db.sh` pass.
- **Backward compatibility notes:** v0.1 `sessions` rows are preserved unchanged. Every v0.1 row was minted and closed inside the same dispatch handler, so all existing rows have `ended_at != NULL` and naturally read as "closed sessions" under v0.2's `ended_at IS NULL = active` predicate. TRUNCATE was deliberately rejected: every `sessions` row is FK-referenced by `audit_events`, and cascade-deleting audit history would violate the append-only audit-trail non-negotiable (GIF-003). Operators wanting to filter analytics to v0.2-style sessions only can use the heuristic `ended_at IS NULL OR (ended_at - started_at) > interval '1 second'` — v0.1 per-call rows had sub-second lifetimes (GIF-021 §C2).

Future SEP-driven additions (SEP-414 W3C Trace Context, SEP-1299 transport-session correlation, per-persona TTL, parent-session linkage for delegation) all land as additive `ALTER TABLE ADD COLUMN` migrations against `gif.sessions`. The repurpose-in-place decision does not foreclose any of them (GIF-021 §C3).

### MCP SDK package layout and version

GIF v0.1 depends on a single npm package, `@modelcontextprotocol/sdk@^1.8.0`. The MCP 2026-07-28 Specification Release Candidate corresponds to a new SDK major version with a **split package layout** (`@modelcontextprotocol/server`, `/client`, `/node`, `/express`, `/hono`, `/fastify`). Each split package corresponds to a different HTTP integration. GIF picks the `@modelcontextprotocol/node` transport because gif's reference server is plain Node `http.createServer` with manual routing — no framework. `/express`, `/hono`, and `/fastify` are alternatives for adopters who run their own MCP server on top of those frameworks; they target the same v2 protocol semantics.

- **v0.1 behavior:** Single import surface. Lockfile pins to a specific `1.x` patch (`1.27.1` at the time of the `v0.2.0-rc.1` cut).
- **v0.2 behavior:** Imports split across `@modelcontextprotocol/server` (low-level `Server` + types + error classes) and `@modelcontextprotocol/node` (`NodeStreamableHTTPServerTransport`). The `v0.2.0-rc.1` tag pins to `2.0.0-alpha.2` (published 2026-04-01 — the latest 2.0 prerelease on npm at cut time; no `rc` dist-tag has been published on the 2.0 line yet). Pin moves to `^2.0.0` at `v0.2.0` final once MCP ships a 2.0 SDK final.

#### Adopter-side package.json changes

GIF's own `package.json` and `mcp-server/package.json` carry the substrate swap. Adopters who maintain their own MCP-server consumer code make the equivalent change in their own repos:

```diff
 "dependencies": {
-  "@modelcontextprotocol/sdk": "^1.8.0"
+  "@modelcontextprotocol/server": "2.0.0-alpha.2",
+  "@modelcontextprotocol/node":   "2.0.0-alpha.2",
+  "@cfworker/json-schema":        "^4.1.1"
 }
```

**`@cfworker/json-schema` is a required peer dependency.** `@modelcontextprotocol/server@2.0.0-alpha.2` declares it as a `peerDependency`, not a regular `dependency`. npm does not auto-install peer deps. Without it, the server boots and immediately throws `ERR_MODULE_NOT_FOUND: Cannot find package '@cfworker/json-schema'` on the first request. The SDK uses cfworker as the default JSON Schema validator for `tools/list` and `tools/call` input schemas. Adopters who pin gif but run their own server hit the same trap and must declare cfworker explicitly. (Upstream commit `434b2f11` later moved the cfworker import to a subpath — adopters on a future SDK release past that commit may no longer need the explicit peer; verify against the SDK version in use.)

**`"type": "module"` is required.** The 2.0 SDK packages are ESM-only (`"exports"` declares only an `import` path, no `require`). Consuming code must declare `"type": "module"` in its `package.json` and use ESM `import` syntax. Adopters whose servers are CJS today need to either convert (typical: switch `tsconfig.json` to `module: NodeNext` + add `"type": "module"`, no source changes needed if imports already use `.js` extensions) or wrap gif consumption through dynamic `import()`. GIF's own source was already written in ESM idiom in v0.1 — the migration was a tsconfig + package.json change with no code rewrites.

#### Import map (old → new)

For adopters maintaining MCP-server consumer code, the v1 → v2 symbol changes are mechanical:

| v0.1 import | v0.2 import |
|---|---|
| `import { Server } from '@modelcontextprotocol/sdk/server/index.js'` | `import { Server } from '@modelcontextprotocol/server'` |
| `import { StreamableHTTPServerTransport } from '@modelcontextprotocol/sdk/server/streamableHttp.js'` | `import { NodeStreamableHTTPServerTransport } from '@modelcontextprotocol/node'` |
| `import { McpError, ErrorCode, ListToolsRequestSchema, CallToolRequestSchema, isInitializeRequest } from '@modelcontextprotocol/sdk/types.js'` | `import { ProtocolError, ProtocolErrorCode, isInitializeRequest } from '@modelcontextprotocol/server'` (request schemas removed — see next row) |
| `server.setRequestHandler(ListToolsRequestSchema, ...)` | `server.setRequestHandler('tools/list', ...)` |
| `server.setRequestHandler(CallToolRequestSchema, ...)` | `server.setRequestHandler('tools/call', ...)` |
| `new McpError(ErrorCode.InvalidParams, msg)` | `new ProtocolError(ProtocolErrorCode.InvalidParams, msg)` |

`StreamableHTTPServerTransport` constructor options (`sessionIdGenerator`, `onsessioninitialized`, `onclose`) are unchanged on `NodeStreamableHTTPServerTransport`. The `handleRequest(req, res, body)` signature is unchanged.

An official codemod for the v1 → v2 migration ships in `@modelcontextprotocol/codemod` (merged upstream as PR #1950, not yet npm-published as of the `v0.2.0-rc.1` cut). Adopters on a later SDK release can run `mcp-codemod v1-to-v2 ./src` to apply the import-map and symbol-rename transforms automatically.

#### SDK version tracking during the RC window

GIF rides the `@modelcontextprotocol/*@2.0.0-alpha.N` (and subsequent `2.0.0-rc.N`) prereleases through the 2026-05-21 → 2026-07-28 RC window. The dependency pin moves to a stable `^2.0.0` constraint only after MCP ships its final spec tag.

**Rationale.** GIF is an early MCP-server implementer for this RC. Tracking the SDK prereleases keeps gif inside the feedback loop while SDK behavior and spec text are still negotiable, and lets gif's own design-pass ADRs respond to SDK-level surprises as they surface.

**Trade-off.** Occasional SDK churn during the RC window: a prerelease may ship a breaking change between alpha numbers, and gif may need a corrective bump. As of the `v0.2.0-rc.1` cut, `2.0.0-alpha.2` has been the only 2.0 line publish for ~8 weeks — the SDK line has been quiet while the spec was finalizing.

#### RC-spec items not yet in `2.0.0-alpha.2`

Three behaviors required by the 2026-07-28 RC spec are not yet present in the SDK at `2.0.0-alpha.2`:

1. **`server/discover` RPC** (SEP-2575, MUST) — servers are required to implement this discovery endpoint. The SDK at alpha.2 does not yet expose it. GIF's tool surface is unaffected; the gap is upstream.
2. **`cacheScope` on `tools/list` results** (SEP-2549, required field on `CacheableResult`) — when this lands, GIF MUST set `cacheScope: 'private'` because GIF's tool list is persona-varying (registry-driven dispatch produces different tool sets per persona). `'public'` or absent would let a shared intermediary serve one persona's governed tool list to another — a scope-leak vector the RC newly introduces.
3. **`ttlMs` + `cacheScope` envelope on list/read results** (SEP-2549) — same SEP, same gating: gates on alpha.3+ landing.

These items are not blocking for the `v0.2.0-rc.1` cut. They become testable, addressable, and certifiable when the SDK upstream lands them (expected in `2.0.0-alpha.3` or `2.0.0-rc.N`). GIF will re-bump and re-verify against the conformance harness when that happens.

**Bug-discovery flow.** Before assuming a symptom is new, check the upstream trackers:

```bash
# SDK behavior, packaging, peer deps, runtime issues
gh issue list --repo modelcontextprotocol/typescript-sdk --search "<symptom>"

# Spec text, SEP clarifications, "the spec says X but does it mean Y"
gh issue list --repo modelcontextprotocol/modelcontextprotocol --search "<symptom>"

# Or search the whole MCP org in one shot
gh search issues "org:modelcontextprotocol <symptom> is:open"
```

If a symptom is not already filed, file it where the bug actually lives — `typescript-sdk` for SDK behavior, `modelcontextprotocol` for spec-level questions or ambiguities.

## Open Questions

GIF-018 enumerated seven open questions (OQ1–OQ7) for the v0.2 design pass. Each is now resolved against the locked D-slot decisions or explicitly carried forward as a follow-up.

- **OQ1 — Minting point.** Resolved by GIF-019 D1(b): new `session_start` MCP tool. Discoverable via `tools/list`; uses the existing persona-validation path; emits a `session_start` audit event.
- **OQ2 — Propagation channel.** Resolved by GIF-019 D2(a): explicit `gif_session_id` argument on every governed tool's `inputSchema.required`. Not `_meta`, not headers. Conformance harnesses assert the field's presence via `tools/list` introspection.
- **OQ3 — Closure semantics.** Resolved by GIF-020: caller-close (`session_close` tool) + hard wall-clock TTL (`GIF_SESSION_TTL_SECONDS`, default 86400), lazy expiry. No inactivity timeout, no background sweeper, no per-persona TTL in v0.2.0. Audit-trail stories for each closure family are documented in GIF-020 §Audit-Trail Story.
- **OQ4 — Identity-binding interaction.** Resolved: strictly orthogonal to GIF-013/GIF-014. The `gif_session_id` handle is a second bearer token whose issuance is gated by the existing `persona_id` identity check; it does not replace, weaken, or wrap `persona_id`. GIF-019 F2 records this explicitly.
- **OQ5 — Contributor session-binding extension.** Resolved: the v0.2 handle is independent of GIF-014's contributor-track structural session binding (which would bind an MCP-transport session to `user_persona_assignments.assignment_id`). Force F9 in GIF-019 captures the forward-compat posture for SEP-1299. CONTRIBUTING.md's "Suggested Projects" entry for structural binding remains accurate; no update is required from v0.2.
- **OQ6 — `_meta.traceparent` co-shipping.** Deferred. SEP-414 propagation has not landed in v0.2.0-rc.1 and remains an opportunistic enhancement; capture would land additively (likely via `gif.sessions.invocation_context` or a dedicated additive column) without re-opening any of D1–D5.
- **OQ7 — MCP Apps coverage.** Deferred. SEP-1865 (sandboxed UI iframes) is not in scope for v0.2.0. A follow-up ADR will specify whether MCP Apps share the same `gif_session_id` as the originating session or mint their own; the v0.2 conformance surface does not currently assert on this surface.

Two additional open items surfaced during the v0.2 design pass and are tracked outside this runbook:

- **Conformance test scenarios.** Promoted to load-bearing for SEP Final per SEP-2484. Coordinated with a separate conformance-specification artifact (out of scope for this repo per GIF-022 §Scope boundary).
- **Wire-level error code names.** Per GIF-020 §What This ADR Does Not Decide and GIF-022 §I2, the names ship with `v0.2.0-rc.1` and may be revised in RC feedback. The audit-event type names (the four enumerated in the rejection-mapping table above) are locked and will not change in the RC window.
