# Migrating from v0.1 to v0.2

> **Status:** Skeleton. Populated incrementally as v0.2 breaking changes land. Treat as authoritative only when `v0.2.0` is tagged.

## Why v0.2 is a Breaking Release

The MCP 2026-07-28 Specification Release Candidate removes the `initialize` handshake and `Mcp-Session-Id` from the protocol core (SEP-2575 + SEP-2567). GIF v0.1's combination-policy enforcement accumulates sensitivity across calls within an MCP transport session that no longer exists in v0.2 of the spec. v0.2 introduces an explicit governance session handle minted and tracked by GIF.

The architectural decisions are recorded in:

- [GIF-018](../../decisions/GIF-018-stateless-mcp-session-handles.md) — scoping ADR (forces, design space, migration constraints).
- [GIF-019](../../decisions/GIF-019-session-handle-mint-and-propagation.md) — first design-pass ADR; locks mint point (`session_start` MCP tool) and propagation channel (explicit `gif_session_id` argument on every governed tool's input schema). Closure semantics, schema impact, and conformance surface are deferred to follow-on ADRs.

### MCP SDK package layout and version

GIF v0.1 depends on a single npm package, `@modelcontextprotocol/sdk@^1.8.0`. The MCP 2026-07-28 Specification Release Candidate corresponds to a new SDK major version with a **split package layout**: `@modelcontextprotocol/server`, `@modelcontextprotocol/node`, and `@modelcontextprotocol/hono` at `2.0.0-alpha.N` (and subsequently `2.0.0-rc.N`).

- **v0.1 behavior:** Single import surface. Lockfile pins to a specific `1.x` patch (currently `1.27.1`).
- **v0.2 behavior:** Imports split across the new package layout. Lockfile pins to whichever prerelease is current at the time of the `v0.2.0-rc.1` tag, transitioning to a stable `^2.0.0` constraint at `v0.2.0` final.
- **Migration steps:** Adopters who maintain their own MCP-server consumer code update their import paths in lockstep with the gif tag they repin to. The exact old-to-new import map is populated here when the GIF-side migration lands in code.

#### SDK version tracking during the RC window

GIF rides the `@modelcontextprotocol/*@2.0.0-alpha.N` (and subsequent `2.0.0-rc.N`) prereleases through the 2026-05-21 → 2026-07-28 RC window. The dependency pin moves to a stable `^2.0.0` constraint only after MCP ships its final spec tag.

**Rationale.** GIF is an early MCP-server implementer for this RC. Tracking the SDK prereleases keeps gif inside the feedback loop while SDK behavior and spec text are still negotiable, and lets gif's own design-pass ADRs respond to SDK-level surprises as they surface.

**Trade-off.** Occasional SDK churn during the RC window: a prerelease may ship a breaking change between alpha numbers, and gif may need a corrective bump.

**Bug-discovery flow.** Before assuming a symptom is new, check the upstream trackers:

```bash
# SDK behavior, packaging, peer deps, runtime issues
gh issue list --repo modelcontextprotocol/typescript-sdk --search "<symptom>"

# Spec text, SEP clarifications, "the spec says X but does it mean Y"
gh issue list --repo modelcontextprotocol/modelcontextprotocol --search "<symptom>"

# Or search the whole MCP org in one shot
gh search issues "org:modelcontextprotocol <symptom> is:open"
```

If a symptom is not already filed, file it where the bug actually lives — `typescript-sdk` for SDK behavior, `modelcontextprotocol` for spec-level questions or ambiguities.

While v0.2 is in RC, adopters can stay on v0.1.x:

```json

```
