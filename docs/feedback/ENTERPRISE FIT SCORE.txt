The 3 Questions I Ask In Every First Meeting
**1. You enforce at MCP, but most enterprise AI tools don't use MCP — they call APIs directly. What happens when my AI app calls the Salesforce API, the ServiceNow API, or a custom internal microservice? Does GIF even see those calls?**

Your enforcement model assumes all tools are registered in your tool registry and invoked through MCP. In reality, enterprise AI apps call dozens of external SaaS APIs and internal services directly via HTTP/REST. Those calls never touch MCP. They bypass your enforcement entirely. You've built governance for a theoretical AI architecture, not the messy reality of how enterprises actually build AI tooling.

If your answer is "adopters need to wrap those API calls as MCP tools," you just admitted this requires rearchitecting every AI integration. That's not infrastructure — that's a rewrite project.

**2. You claim structural immutability via INSERT-only permissions, but Postgres superusers can ALTER TABLE, DROP TABLE, and pg_dump/pg_restore can reconstruct history. How do you prevent a compromised superuser or a motivated DBA from rewriting the audit log?**

This is the core of your moat and I don't believe it survives real-world operational requirements. When I need to migrate databases, patch Postgres, recover from corruption, or respond to a ransomware incident, I'm using superuser credentials and backup tooling. If any of those paths allow audit log mutation — even unintentionally — your "structural guarantee" is a policy, not a structure.

What I need to hear: cryptographic signing of each audit record with tamper-evident logs published to an external witness (e.g., append-only S3 with object lock, or a blockchain-style commitment). INSERT-only permissions are not sufficient.

**3. When a persona is revoked, you claim all active sessions terminate atomically. How does that work when the AI agent is mid-execution inside a tool that takes 30 seconds to complete — do you kill the process, let it finish, or leave it in an undefined state?**

Session termination sounds clean in theory. In practice, if an AI agent is halfway through writing data to a database, calling an external API, or processing a file, and I revoke the persona, what happens? If you kill the process, I've got partial writes and data corruption. If you let it finish, the "atomic revocation" isn't atomic. If you leave it undefined, I've got a governance gap.

This is the kind of edge case that every IAM vendor handwaves until it causes a production incident. I need to know you've thought through the state machine.
